## Рекомендуемый порядок следующих шагов

### Шаг 1. Ручная приемка

Сначала пройти все ключевые вкладки в desktop и dev-режиме. Это сейчас самый важный незакрытый блок, потому что сборка уже зеленая, а реальные UX-регрессии ловятся именно руками.

### Шаг 2. Финальная зачистка текста

После ручной приемки собрать последние остатки по нечитаемым строкам и сразу централизовать UI-лейблы, чтобы новые правки не возвращали проблему.

### Шаг 3. Усиление launcher-сценариев

Следом стоит сделать `Elira.bat` максимально предсказуемым: понятные сообщения, проверка зависимостей, ясный сценарий старта backend/frontend/Tauri.

### Шаг 4. Централизованное логирование

После стабилизации запуска стоит добавить единый logging-контур. Это даст нормальную диагностику проблем в Tauri, backend, Telegram и pipeline-задачах без ручного поиска по разрозненным местам.

### Шаг 5. Финальная cleanup-волна

Только после стабилизации рантайма имеет смысл идти в глубокий cleanup legacy `Jarvis` следов, переименование и косметическую зачистку всей кодовой базы.

## Пожелания и рекомендации

### 1. Не держать route-строки в компонентах

Все backend вызовы должны оставаться только в `frontend/src/api/*`. Это уже правильное направление, его нужно сохранить как правило.

### 2. Свести пользовательские подписи в одно место

Лучше вынести UI-строки в централизованный словарь или constants-модуль. Это уменьшит риск повторной поломки кодировки и ускорит поддержку.

### 3. Не редактировать русский UI текст через сомнительные кодировки

Если правки идут из среды с проблемной консольной кодировкой, лучше писать через нормальный UTF-8 редактор или автоматизированный safe rewrite. Иначе битые байты возвращаются очень легко.

### 4. Держать capability-status как часть UX

Capability warnings полезны и для пользователя, и для отладки. Их не стоит скрывать, особенно если проект должен работать частично даже без optional dependencies.

### 5. Перед следующей большой волной добавить короткий release checklist

Минимум:
- backend импортируется;
- OpenAPI строится;
- frontend собирается;
- launcher стартует;
- ключевые вкладки открываются;
- unreadable text отсутствует.

### 6. Логирование должно стать частью базовой инфраструктуры

Сейчас в проекте уже есть полезные куски логирования, но они разбросаны по сервисам. Следующая правильная цель не просто “логировать больше”, а собрать единый контур:
- access log;
- audit log;
- file log;
- безопасная фильтрация секретов;
- понятная точка, где разработчик смотрит причину ошибки.
