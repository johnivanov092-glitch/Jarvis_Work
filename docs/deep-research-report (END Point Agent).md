### Главные риски, которые надо закрыть до “семейного multi‑user”

Проблемы не про «модель», а про **контуры выполнения действий**:

Терминал как сейчас — это по факту “remote code execution” в пределах workspace. Для одного доверенного пользователя это ок, для 10–20 пользователей даже дома нужно: (а) аутентификация, (б) пер‑юзер изоляция, (в) роли и разрешения, (г) audit‑лог. fileciteturn11file0L1-L1

Плагины как исполняемый Python без песочницы — тот же класс риска (и даже хуже, потому что плагины могут жить долго и выполняться хуками). fileciteturn12file0L1-L1

`.gitignore` сейчас не отражает вашу «единую data‑root в `data/`» как рабочий контур: он игнорирует `backend/data/*`, но не гарантирует игнорирование **`data/` в корне** (а это то, куда вы фактически складываете runtime‑БД/загрузки). Это прямой риск случайно закоммитить личные документы/историю чатов/ключи. fileciteturn10file0L1-L1

Вывод: переход на vLLM и железо даст “производительность”, но **реальным узким местом станет безопасность выполнения инструментов и управление данными пользователей**.

## Железо и дисковая раскладка под vLLM + Plex

### GPU и общий баланс ресурсов

RTX 5090 действительно заявляется как карта с **32 ГБ GDDR7**. citeturn4search2  
Для “домашнего Codex/Opus‑класса” это означает:

Одна RTX 5090 32GB — отличный базовый уровень для инференса крупных 14B–32B моделей (в зависимости от квантования/контекста/параллелизма), а также для стабильной работы vLLM с приличной concurrency.

Две RTX 5090 (если реально пойдёте в 2×) дают два сценария:
- грузить **более крупные модели**, распределяя веса по GPU (tensor/pipeline parallel),
- или держать **две реплики** (data parallel / две инстанции сервера) ради параллельных запросов.

vLLM официально поддерживает multi‑GPU serving через `--tensor-parallel-size` и также описывает pipeline parallelism. citeturn5search2turn5search7  
Важная практическая оговорка из доков vLLM: при отсутствии NVLink на “обычных” GPU может быть выгоднее pipeline‑parallel, чем tensor‑parallel. citeturn5search7

### RAM: 64 vs 128 ГБ

Ваше “64 минимум, лучше 128” — технически оправдано. fileciteturn0file0  
Причины (прагматично):
- vLLM имеет параметры CPU‑offload и swap‑space для KV‑кэша/памяти, и в целом больше RAM расширяет “комфортную зону” по контексту и параллельным запросам. citeturn5search0  
- RAG/индексация (embeddings, FAISS/Qdrant) и большие документы (PDF, кодовые базы) тоже любят RAM.

Если бюджет позволяет — 128 ГБ даёт запас под «эволюцию», особенно когда появится multi‑user и фоновые пайплайны.

### NVMe 1 vs 2 (и вопрос “лучше один большой или разделять”)

Ваше предложение:
- NVMe1: system + docker
- NVMe2: модели / Hugging Face cache / vLLM cache
— это **правильная базовая архитектура** для предсказуемости I/O и обслуживания. fileciteturn0file0

Почему разделение физически полезнее, чем один NVMe:
- меньше конкуренции за IOPS между Docker layers/логами и тяжёлым чтением весов;
- проще переустановка ОС/докера без риска снести модели/кэш;
- проще мониторить износ и планировать замену.

Когда можно оставить один “очень большой и быстрый NVMe”:
- если вы точно хотите минимизировать число устройств, и готовы дисциплинированно разнести: `/` + `/var/lib/docker` + `/srv/models` по разным разделам/томам, и принять, что любое большое обновление контейнеров может влиять на latency инференса.

### Hugging Face cache и путь на NVMe2

Документация `huggingface_hub` описывает `HF_HOME` (куда кладутся токен и кэш) и `HF_HUB_CACHE` (кэш репозиториев моделей/датасетов) — это ровно то, что нужно закрепить на NVMe2. citeturn2search7  
А официальный docker‑пример vLLM прямо монтирует `~/.cache/huggingface` внутрь контейнера. citeturn0search8

Практический стандарт для сервера:
- `/srv/ai/cache/huggingface` → `HF_HOME=/srv/ai/cache/huggingface`
- `/srv/ai/models` (если храните локальные зеркала моделей/весов отдельно от hub‑кэша)

### Plex media: отдельный массив

Ваше требование «HDD/SSD массив отдельно под media для Plex» — верное. fileciteturn0file0  
Рекомендация по надёжности: 2×10 ТБ лучше собирать не в «один диск», а в массив с понятной стратегией отказа (минимум зеркало), иначе вы получите “20 ТБ” ценой риска потери всей медиатеки при отказе одного диска.

## Базовая инфраструктура на Ubuntu Server 24.04 + Docker Compose + NVIDIA

### NVIDIA Container Toolkit

Официальная инструкция NVIDIA для Container Toolkit показывает типовой путь: поставить пакет через репозиторий и затем выполнить `nvidia-ctk runtime configure --runtime=docker`, после чего перезапустить Docker. citeturn2search0  
Это обязательный шаг, чтобы vLLM контейнер видел GPU.

### vLLM как отдельный контейнер

vLLM имеет официальный docker‑образ `vllm/vllm-openai` и рекомендуемый флаг `--ipc=host` (а также монтирование HF‑кэша). citeturn0search8  
vLLM также документирует “OpenAI‑Compatible Server”: запуск `vllm serve ...` и возможность обращаться к нему OpenAI‑клиентом через `base_url=http://localhost:8000/v1`. citeturn0search3

Критичные настройки vLLM, которые вам пригодятся на практике:
- `--gpu-memory-utilization` (по умолчанию 0.9) — ограничение доли VRAM под конкретный vLLM‑инстанс; полезно, если делите GPU между сервисами/инстансами. citeturn5search0
- `--tensor-parallel-size` / `--pipeline-parallel-size` для 2×GPU сценариев. citeturn5search2turn5search7

### Plex как отдельный контейнер

Отделение Plex — правильно (и организационно, и по обновлениям). fileciteturn0file0  
Ключевой момент — **ресурсные лимиты** (CPU/RAM/IO) и политика доступа к GPU (если вы планируете NVENC/транскодирование). В домашнем стеке важно, чтобы Plex не “съедал” GPU в момент, когда вы хотите интерактивную работу агента.

## План перехода на vLLM и адаптация под Ollama / vLLM / LM Studio

### Почему “OpenAI‑совместимый слой” — ваш лучший абстрактный контракт

vLLM уже предоставляет OpenAI‑совместимый API. citeturn0search3  
LM Studio тоже предоставляет OpenAI‑совместимые эндпоинты (`GET /v1/models`, `POST /v1/chat/completions`, `POST /v1/embeddings`, `POST /v1/completions`). citeturn1search2

Это означает, что **самый практичный способ поддерживать разные рантаймы** — сделать в Elira единый интерфейс “Chat/Embeddings provider” и две реализации:
- `OllamaProvider` (то, что у вас сейчас),
- `OpenAICompatibleProvider` (vLLM, LM Studio).

Важный плюс: в UI можно не хардкодить модели, а подхватывать их из `/v1/models` там, где это возможно. citeturn1search2

### Что менять в проекте (конкретно)

Минимально необходимое (с высокой отдачей):
- Вынести текущую “ollama‑логику” из `core/llm.py` в модуль провайдера, а основной код перевести на интерфейс вида `provider.chat(...) / provider.stream_chat(...) / provider.list_models()`.
- Добавить конфиг: `LLM_BACKEND=ollama|openai_compat`, `OPENAI_BASE_URL`, `OPENAI_API_KEY` (для локальных серверов ключ часто “фиктивный”, но клиентам нужен).
- Для vLLM: добавить отдельный контейнер, и в Elira‑backend ходить на него по внутренней docker‑сети.

Второй слой (следующий шаг):
- Embeddings вынести в отдельный сервис/провайдер (часто выгодно держать embeddings модель отдельно от “болтливой” модели).

## Агентная эволюция, плагины, OS‑Copilot, Open Interpreter, LangChain/LlamaIndex, MCP и Computer Use

### OS‑Copilot контур: что у вас уже есть и что добавить

У вас уже есть два “акта действия”:
- терминал, который реально выполняет команды (сейчас — без полноценной auth‑модели), fileciteturn11file0L1-L1
- плагины как исполняемые Python‑модули. fileciteturn12file0L1-L1

Для “DevOps‑помощника уровня Codex” не хватает трёх вещей:
- строгой модели разрешений (policy),
- песочниц (sandbox) для кода/команд/плагинов,
- трассировки и оценивания качества (evals).

### Open Interpreter: почему интересен, но как использовать безопасно

Open Interpreter прямо признаёт, что запуск LLM‑сгенерированного кода рискован, и предлагает Safe Mode (сканирование, отключение auto‑run) — но подчёркивает, что гарантий безопасности нет. citeturn2search2turn2search10  
Также у него есть экспериментальная интеграция с Docker для изоляции исполнения. citeturn3search1

Практическое применение в вашем проекте:
- не встраивать Open Interpreter “как есть” в прод‑контур,
- а перенять его идеи: **интерактивное подтверждение действий**, **сканирование/линт** (semgrep), и **исполнение в контейнере**, а не на хосте.

### LangChain / LlamaIndex: что выбрать для вашего случая

LangChain (через LangGraph) делает акцент на production‑агентах как графе узлов (модель, tools, middleware), с циклом “пока не stop condition”. citeturn1search11  
LlamaIndex формализует агента как связку LLM+memory+tools и даёт AgentWorkflow/FunctionAgent как оркестратор (включая multi‑agent) и простой путь описывать tools. citeturn3search5turn3search0

Рекомендация «чётко по делу»:
- Если вы хотите **быстро усилить tool‑loop и наблюдаемость** — LangChain/LangGraph хорош как готовый runtime каркас. citeturn1search11  
- Если вы хотите **сильный контур документов/RAG/памяти** и “document‑agents” — LlamaIndex обычно ближе. citeturn3search5turn3search0

Но: у вас уже есть существенная собственная агентная логика. Поэтому лучший путь — не переписывать всё на LangChain/LlamaIndex, а:
- оставить ваш backend как “Agent OS”,
- подключить LangChain/LlamaIndex **точечно**: либо как библиотеку для tool selection/structured output, либо как framework для отдельных режимов (например, “Документ‑агент”, “RAG‑агент”).

### Anthropic Computer Use и как сделать локальный аналог

Anthropic описывает Computer Use tool как механизм, который даёт модели **скриншот + управление мышью/клавиатурой**, и отдельно подчёркивает меры безопасности: использовать выделенную VM/контейнер, не давать доступ к чувствительным данным и т.п. citeturn0search1turn0search11

Локальный аналог для вас (без облака):
- поднять “рабочий стол” в отдельной песочнице (контейнер/VM) с VNC/NoVNC,
- добавить tool‑API: `screenshot()`, `mouse(action)`, `keyboard(type/shortcut)`,
- и разрешать это только роли “admin” или по подтверждению.

### MCP (Model Context Protocol): зачем вам

Anthropic определяет MCP как открытый протокол “стандартизации подключения контекста и инструментов к LLM” (метафора USB‑C для AI‑приложений). citeturn0search0

Для вашего проекта MCP полезен не “для моды”, а как способ:
- стандартизировать подключение внешних tools (домашние сервисы, NAS, Git, календарь, monitoring),
- отделить “Agent OS” от “tool servers”.

Практический вывод: делайте 2 контрагента:
- Elira backend = MCP‑client (потребляет tools),
- ваши интеграции = MCP‑servers (каждый со своей политикой доступа и логированием).
